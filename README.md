# WatchShopFx

`WatchShopFx` - учебное JavaFX-приложение "магазин часов".
Проект показывает, как связать UI (FXML + Controllers), модель данных и бизнес-логику с автоматическим обновлением экрана через наблюдателей.

## Для чего проект

Проект полезен, если вы хотите:

- разобраться, как устроено JavaFX-приложение с несколькими экранами/панелями;
- увидеть на практике разделение на Model и Controller;
- понять, как сделать "живую" статистику, которая обновляется после каждого действия;
- потренировать базовые принципы ООП и паттерны проектирования.

## Что умеет приложение

Приложение поддерживает:

- отображение списка часов в магазине;
- добавление новых часов (бренд, стартовое время, стоимость);
- продажу часов (удаление из списка);
- установку времени для одних часов и массово для всех часов;
- расчет и показ статистики:
- количество часов по брендам;
- список часов по убыванию стоимости;
- самые дорогие часы;
- общие показатели магазина (капитал, продано штук, продано на сумму, ошибки времени).
- консоль событий (лог пользовательских действий и ошибок).

## Структура проекта

Основные директории:

- `src/main/java/org/example/watchshopfx` - ядро приложения;
- `src/main/java/org/example/watchshopfx/Models` - модели данных и бизнес-логика;
- `src/main/java/org/example/watchshopfx/Controllers` - контроллеры UI;

## Архитектура и поток данных

Приложение построено вокруг трех моделей:

- `ShopModel` - инвентарь часов, выборки и статистика;
- `StoreStateModel` - состояние магазина (капитал, продажи, ошибки времени);
- `ConsoleModel` - список сообщений (лог).

Контроллеры подписываются на нужные модели через `IObserver`.
Когда модель меняется, она вызывает `eventCall()`, и подписанные контроллеры перерисовывают свой кусок интерфейса.

Типичный поток:

1. Пользователь нажал кнопку в UI.
2. Контроллер вызвал метод модели.
3. Модель изменила состояние и уведомила наблюдателей.
4. Наблюдатели обновили текст/таблицы/списки в UI.

## Паттерны, которые используются

### 1) Observer (Наблюдатель)

Где:

- интерфейс `IObserver`;
- у моделей есть списки наблюдателей (`allObserver`);
- при изменении вызывается `eventCall()`.

Зачем:

- убрать жесткую связку "модель -> конкретный контроллер";
- автоматически синхронизировать UI с данными.

### 2) Singleton-подобный доступ через BModels

Где:

- `BShopModel`, `BConsoleModel`, `BStoreStateModel`.

Как работает:

- каждый `B*Model` хранит один `static final` экземпляр модели;
- метод `build()` возвращает этот экземпляр всем контроллерам.

Зачем:

- все контроллеры работают с единым состоянием, а не с копиями.

### 3) MVC-подход (практический вариант для JavaFX)

- `Model` - `ShopModel`, `StoreStateModel`, `ConsoleModel`;
- `View` - `FXML` (`MainView.fxml` и `Views/*.fxml`);
- `Controller` - классы в `Controllers`.

Зачем:

- разделение ответственности:
- бизнес-логика не смешивается с визуальной разметкой;
- UI легче менять без переписывания модели.

## ООП-парадигмы с примерами из кода

### 1) Инкапсуляция

Пример:

- поля `Watch` (`brand`, `cost`, `hours`, `minutes`) скрыты (`private`);
- менять время можно только через `setTime`, где есть валидация формата и диапазона.

Польза:

- нельзя "сломать" состояние объекта, записав некорректные значения напрямую.

### 2) Абстракция

Пример:

- внешний код работает с методами `ShopModel` (`addWatch`, `getBrandStatisticsDesc`, `setTimeForAll`);
- детали хранения (`ArrayList`, `Map`, сортировки) скрыты внутри класса.

Польза:

- контроллеры проще и читаемее;
- можно менять внутреннюю реализацию модели без изменения UI-слоя.

### 3) Полиморфизм

Пример:

- единый интерфейс `IObserver` с перегруженными `event(...)`;
- разные контроллеры реализуют только нужные обработчики событий.

Польза:

- одинаковый контракт подписки для разных типов моделей;
- расширяемость: можно добавить новый контроллер-наблюдатель без изменения моделей.

### 4) Наследование

В проекте почти не используется классическое наследование бизнес-классов.
Основной акцент сделан на композицию (модель + контроллер + интерфейс наблюдателя), что для JavaFX-приложений обычно практичнее.

## Подробный справочник по классам, полям и методам

Ниже - практический "мини-справочник", чтобы быстро понять, за что отвечает любая важная переменная.

### `App`

Роль: точка входа JavaFX.

Ключевые методы:

- `start(Stage stage)`:
- загружает `MainView.fxml`;
- создает `Scene`;
- задает заголовок окна;
- показывает окно.
- `main(String[] args)`:
- запускает JavaFX runtime через `launch()`.

### `Watch`

Роль: доменная сущность "часы".

Поля:

- `brand` (`String`, `final`) - бренд часов. Устанавливается один раз в конструкторе.
- `cost` (`int`, `final`) - стоимость часов. Устанавливается один раз в конструкторе.
- `hours` (`int`) - часы текущего времени на циферблате.
- `minutes` (`int`) - минуты текущего времени.

Ключевые методы:

- `Watch(String brand, int cost, String initialTime)`:
- валидирует бренд и стоимость;
- парсит и устанавливает начальное время.
- `getBrand()` - вернуть бренд.
- `getCost()` - вернуть стоимость.
- `getTimeString()` - вернуть время в формате `HH:mm`.
- `setTime(String time)` - установить новое время с валидацией.
- `parseTime(String time)`:
- проверка `null`;
- проверка формата `\d{2}:\d{2}`;
- проверка диапазонов `0..23` и `0..59`.
- `equals/hashCode` - сравнение по бренду, стоимости и времени.

Когда смотреть этот класс:

- если нужно понять, почему падает установка времени;
- если нужно изменить формат/правила валидации времени.

### `InvalidWatchTimeException`

Роль: checked-exception для ошибок пользовательского времени.

Зачем:

- отделяет ошибки валидации времени от других ошибок (`NumberFormatException`, `IllegalArgumentException`).

### `IObserver`

Роль: контракт для подписчиков на изменения моделей.

Методы:

- `event(ShopModel m)`;
- `event(ConsoleModel c)`;
- `event(StoreStateModel e)`.

Примечание:

- контроллеры обычно реализуют все три метода, но часть оставляют пустыми.

### `ShopModel`

Роль: центральная модель инвентаря часов + статистика/выборки.

Поля:

- `KNOWN_BRANDS` (`List<String>`, `static final`) - предустановленный список брендов для подсказок в UI.
- `allWatches` (`ArrayList<Watch>`) - весь текущий инвентарь магазина.
- `allObserver` (`ArrayList<IObserver>`) - подписчики на изменения инвентаря.

Конструктор:

- добавляет стартовые часы через `addInitialWatch(...)`.

Базовые методы управления:

- `countWatches()` - количество часов в наличии.
- `totalInventoryCost()` - суммарная стоимость всего инвентаря.
- `addObserver(IObserver observer)` - подписка наблюдателя и немедленное оповещение.
- `addWatch(Watch watch)` - добавить часы и оповестить подписчиков.
- `removeWatch(Watch watch)` - удалить часы и оповестить подписчиков.
- `notifyChanged()` - ручное оповещение (например, после `watch.setTime(...)` вне самой модели).
- `setTimeForAll(String time)` - массовая установка времени для всех часов.

Методы статистики/выборок:

- `getMostExpensiveWatches()` - вернуть список самых дорогих часов (если несколько с одинаковой максимальной ценой, вернутся все).
- `getBrandStatisticsDesc()` - вернуть пары `бренд -> количество`, отсортированные:
- сначала по убыванию количества;
- при равенстве - по имени бренда (без учета регистра).
- `getUniqueBrandsSorted()` - бренды из инвентаря без дублей, отсортированы.
- `getBrandSuggestionsSorted()`:
- объединяет `KNOWN_BRANDS` + реальные бренды из инвентаря;
- удаляет дубли;
- сортирует без учета регистра.
- `getWatchesByCostDesc()` - часы, отсортированные по стоимости по убыванию.

Служебные методы:

- `eventCall()` - вызывает `event(this)` у всех подписчиков.
- `addInitialWatch(...)` - безопасно добавляет стартовые часы (ошибка времени тут считается ошибкой конфигурации и превращается в `IllegalStateException`).
- `iterator()` - позволяет итерировать `ShopModel` через `for (Watch w : model)`.

### `StoreStateModel`

Роль: финансово-операционная статистика магазина.

Поля:

- `INITIAL_CAPITAL` (`100_000`) - стартовый капитал.
- `allObserver` - подписчики на изменение состояния магазина.
- `invalidTimeErrors` - сколько раз пользователь ввел некорректное время.
- `soldCount` - сколько часов продано.
- `soldAmount` - на какую сумму продано.
- `cashBalance` - текущий капитал (стартовый - закупки + продажи).

Методы:

- `onWatchAdded(Watch watch)` - уменьшает капитал на стоимость закупки.
- `onWatchSold(Watch watch)` - увеличивает капитал и счетчики продаж.
- `onInvalidTimeInput()` - увеличивает счетчик ошибок времени.
- `addObserver(...)` и `eventCall()` - механизм оповещений.

### `ConsoleModel`

Роль: хранение и рассылка консольных сообщений.

Поля:

- `messages` - список сообщений (новые добавляются в начало).
- `allObserver` - подписчики.

Методы:

- `addMessage(String message)` - добавить сообщение в начало и обновить UI.
- `getMessages()` - вернуть неизменяемую копию сообщений.
- `clear()` - очистить лог.
- `addObserver(...)`/`eventCall()` - подписка и оповещение.

### `BModels` (`BShopModel`, `BConsoleModel`, `BStoreStateModel`)

Роль: дать всем контроллерам один и тот же экземпляр модели.

Поля:

- `private static final ... model` - единственный экземпляр.

Метод:

- `build()` - возвращает этот экземпляр.

### Контроллеры

#### `MainViewController`

Роль:

- собирает главный экран из вложенных `FXML`-панелей;
- отображает список часов справа;
- подписан на `ShopModel` и перерисовывает карточки часов при изменениях.

Поля:

- `shopModel` - доступ к инвентарю;
- `main` - корневой `GridPane`;
- `watchListGrid` - контейнер списка карточек часов.

#### `AddWatchViewController`

Роль: форма добавления часов.

Поля UI:

- `brandComboBox` - выбор/ввод бренда;
- `timeField` - начальное время;
- `costField` - стоимость;
- `addButton` - кнопка добавления.

Логика `add()`:

1. Читает бренд из editable-комбобокса.
2. Проверяет, что бренд не пустой.
3. Парсит стоимость.
4. Проверяет, хватает ли капитала.
5. Создает `Watch`, добавляет в `ShopModel`.
6. Обновляет `StoreStateModel` (`onWatchAdded`).
7. Пишет сообщение в `ConsoleModel`.
8. При ошибках пишет понятное сообщение и при ошибке времени увеличивает счетчик `invalidTimeErrors`.

#### `WatchViewController`

Роль: управление одной карточкой часов в списке.

Поля UI:

- `brandLabel`, `costLabel`, `timeLabel` - отображение данных часов;
- `setTimeField` - поле установки времени.

Ключевая логика:

- `remove()`:
- удаляет часы из `ShopModel`;
- обновляет продажи в `StoreStateModel`;
- пишет лог в `ConsoleModel`.
- `setTime()`:
- пытается установить время конкретным часам;
- вручную вызывает `shopModel.notifyChanged()`, чтобы обновить экран;
- логирует результат или ошибку.

#### `BulkTimeUpdateViewController`

Роль: массовая установка времени для всех часов.

Поле:

- `allTimeField` - ввод времени.

Логика:

- вызывает `shopModel.setTimeForAll(...)`;
- при ошибке времени увеличивает счетчик ошибок и пишет в лог.

#### `StatisticsViewController`

Роль: вывод статистики количества часов по брендам.

Поля:

- `brandStatsBox` - контейнер, куда кладется текст-таблица;
- `monoFont` - моноширинный шрифт для ровного выравнивания колонок.

Данные берутся из `shopModel.getBrandStatisticsDesc()`.

#### `InventoryCostStatisticsViewController`

Роль: вывод инвентаря, отсортированного по убыванию цены.

Поле:

- `inventoryStatsBox` - контейнер для таблицы.

Данные берутся из `shopModel.getWatchesByCostDesc()`.

#### `MostExpensiveWatchViewController`

Роль: вывод самых дорогих часов.

Поле:

- `expensiveList` - контейнер списка.

Данные берутся из `shopModel.getMostExpensiveWatches()`.

#### `ShopStatusViewController`

Роль: панель общих KPI магазина.

Поля UI:

- `watchCountLabel` - количество часов;
- `inventoryCostLabel` - сумма стоимости инвентаря;
- `cashBalanceLabel` - текущий капитал;
- `invalidTimeErrorsLabel` - количество ошибок времени;
- `soldCountLabel` - количество проданных часов;
- `soldAmountLabel` - сумма продаж.

Особенность:

- подписывается сразу на `ShopModel` и `StoreStateModel`, потому что часть метрик живет в разных моделях.

#### `ConsoleViewController`

Роль: отображение логов из `ConsoleModel`.

Поля:

- `consoleMessagesBox` - контейнер для текстового лога;
- `monoFont` - шрифт вывода.

## Подробно: методы отбора и расчета статистики в ShopModel

Ниже - ключевой раздел по вашему запросу.

### 1) `getMostExpensiveWatches()`

Что делает:

- находит максимальную стоимость в текущем списке часов;
- возвращает все часы, у которых стоимость равна максимуму.

Почему так:

- если есть несколько одинаково дорогих моделей, нельзя терять данные, поэтому возвращается список, а не один объект.

Алгоритм:

1. Если список пустой -> вернуть пустой список.
2. Найти `maxCost` через stream `max()`.
3. Отфильтровать `allWatches` по `watch.getCost() == maxCost`.
4. Собрать в список.

Сложность:

- время `O(n)`;
- память `O(k)`, где `k` - количество самых дорогих часов.

Где используется:

- `MostExpensiveWatchViewController`.

### 2) `getBrandStatisticsDesc()`

Что делает:

- считает, сколько часов каждого бренда есть в магазине;
- возвращает список пар `Map.Entry<String, Integer>` с сортировкой.

Сортировка:

- сначала по количеству по убыванию;
- если количество одинаковое, по названию бренда (регистронезависимо, A-Z).

Алгоритм:

1. Создать `Map<String, Integer> counts`.
2. Пройти по всем часам:
- увеличить счетчик бренда на 1 (`getOrDefault(..., 0) + 1`).
3. Преобразовать `entrySet()` в stream.
4. Отсортировать кастомным компаратором:
- `Integer.compare(b.getValue(), a.getValue())`;
- потом `a.getKey().compareToIgnoreCase(b.getKey())`.
5. Вернуть `List<Entry<...>>`.

Сложность:

- подсчет `O(n)`;
- сортировка `O(m log m)`, где `m` - число уникальных брендов.

Где используется:

- `StatisticsViewController`.

### 3) `getWatchesByCostDesc()`

Что делает:

- возвращает копию списка часов, отсортированную по цене по убыванию.

Алгоритм:

1. Берет stream из `allWatches`.
2. Сортирует `Comparator.comparingInt(Watch::getCost).reversed()`.
3. Собирает в новый список.

Сложность:

- `O(n log n)`.

Где используется:

- `InventoryCostStatisticsViewController`.

### 4) `totalInventoryCost()`

Что делает:

- считает общую стоимость текущего инвентаря.

Алгоритм:

- `allWatches.stream().mapToInt(Watch::getCost).sum()`.

Сложность:

- `O(n)`.

Где используется:

- `ShopStatusViewController`.

### 5) `countWatches()`

Что делает:

- возвращает количество часов в магазине.

Алгоритм:

- `allWatches.size()`.

Сложность:

- `O(1)`.

Где используется:

- `ShopStatusViewController`.

### 6) `getUniqueBrandsSorted()`

Что делает:

- возвращает уникальные бренды из фактического инвентаря, отсортированные лексикографически.

Алгоритм:

1. Взять `brand` у всех часов.
2. Положить в `TreeSet` (он сам удаляет дубли и сортирует).
3. Преобразовать обратно в `ArrayList`.

Сложность:

- примерно `O(n log n)`.

### 7) `getBrandSuggestionsSorted()`

Что делает:

- формирует список подсказок брендов для формы добавления:
- включает "известные бренды" (`KNOWN_BRANDS`);
- плюс бренды, которые уже фактически есть в магазине.

Алгоритм:

1. Создать `LinkedHashSet` на базе `KNOWN_BRANDS`.
2. Добавить туда текущие уникальные бренды.
3. Отсортировать результат `String.CASE_INSENSITIVE_ORDER`.

Почему это удобно:

- пользователь видит готовые бренды;
- при этом система не "забывает" кастомные бренды, введенные вручную.

Где используется:

- `AddWatchViewController.refreshBrandOptions()`.

## Сценарии использования

### Добавление часов

1. Ввод бренда, времени и стоимости.
2. Нажатие "Добавить в магазин".
3. Создание `Watch`.
4. `ShopModel.addWatch(...)`.
5. `StoreStateModel.onWatchAdded(...)` (уменьшается капитал).
6. `ConsoleModel.addMessage(...)`.
7. Наблюдатели обновляют UI.

### Продажа часов

1. На карточке часов нажимается `$`.
2. `ShopModel.removeWatch(...)`.
3. `StoreStateModel.onWatchSold(...)`.
4. `ConsoleModel.addMessage(...)`.
5. UI обновляется.

### Массовая установка времени

1. Ввод времени в блоке "для всех часов".
2. `ShopModel.setTimeForAll(...)`.
3. При успехе логируется событие.
4. При ошибке:
- увеличивается `invalidTimeErrors`;
- в консоль добавляется сообщение об ошибке.

## Ограничения и важные нюансы

- Валюта стоимости в коде не выделена явно (просто `int`).
- Потокобезопасность не реализована: модели используют обычные `ArrayList` (для учебного desktop-сценария этого обычно достаточно).
- Логика уведомлений синхронная: если наблюдатель выбросит исключение, это может повлиять на общий поток обновления.
- В `ShopModel.addObserver(...)` есть немедленный вызов `eventCall()`: UI сразу получает initial render.

## Идеи для улучшения

1. Вынести форматирование таблиц из контроллеров в отдельный helper/service.
2. Добавить тесты для `ShopModel` (особенно сортировки и подсчеты статистики).
3. Хранить деньги в `BigDecimal`, если нужна точная финансовая модель.
4. Добавить локализацию строк UI (`ResourceBundle`) вместо строк в коде/FXML.
5. Добавить фильтры статистики по диапазонам цены или бренду.
